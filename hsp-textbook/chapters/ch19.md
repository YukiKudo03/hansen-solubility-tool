**Chapter 19: Software Architecture and Design Principles for HSP Computation Tools**

> **概要:** 本章では、プロダクショングレードのHSP評価ツールを支えるソフトウェアアーキテクチャと設計原則を解説する。対象システムはElectron・React・TypeScriptを技術基盤とし、117のコアモジュールからなる純粋関数ドメインロジック、Repositoryパターンによるデータアクセス抽象化、contextBridgeを介したElectron IPC通信設計、そして2604件のユニットテスト（カバレッジ98.88%）と295件のE2Eテスト（Playwright）による包括的テスト駆動開発を実現している。科学計算ソフトウェアの保守性・拡張性・正確性をいかに設計レベルで担保するかという、研究開発の現場で広く応用可能な方法論を提示する。

## 19.1 科学計算ソフトウェアのアーキテクチャ要件

### 19.1.1 歴史的背景

科学計算ツールのアーキテクチャは1990年代以降、劇的な進化を遂げてきた。初期のHSPソフトウェア——Hansenが自ら開発に関わったHSPiP（初版2001年頃）[Hansen, 2007]——はモノリシックなデスクトップアーキテクチャを採用していた。計算ロジック、データアクセス、UIが密結合しており、個別のテストや独立した改修が困難であった。

2010年代に入ると、ウェブ技術のデスクトップ応用が急速に進展した。GitHub社が2013年にリリースしたElectronフレームワーク [Electron Contributors, 2013] はNode.jsとChromiumを統合し、クロスプラットフォームデスクトップアプリケーションの開発を劇的に効率化した。Facebook社（現Meta）が同年発表したReact [Facebook, 2013] はコンポーネントベースUI設計を普及させ、Microsoft社のTypeScript（2012年）[Microsoft, 2012] はJavaScriptに静的型付けを導入した。

これらの技術革新を背景に、科学計算ソフトウェアにもクリーンアーキテクチャの原則 [Martin, 2017] を適用する道が開かれた。ドメイン駆動設計（DDD）のRepositoryパターン [Evans, 2003; Fowler, 2002] やテスト駆動開発（TDD）[Beck, 2002] といったソフトウェア工学の成熟した手法を、HSP計算のような領域固有の数値計算に組み込むことで、保守性・拡張性・正確性を高い水準で両立させることが可能となった。

### 19.1.2 HSP計算ツール固有の要求

HSP評価ツールには、一般的なビジネスアプリケーションとは異なる固有の品質要求が存在する。

**計算の正確性と再現性.** $R_a$ やREDの計算結果は文献値と一致しなければならない。浮動小数点誤差の蓄積、状態依存のバグ、外部環境への依存はすべて信頼性を損なう要因である。

**多様な評価パイプライン.** 本システムは93以上の評価タブを擁し、溶媒スクリーニング、球面フィッティング、グループ寄与法推定、ブレンド最適化、分散剤評価など多岐にわたるパイプラインを提供する。これらは共通の基盤の上に独立して追加・修正できなければならない。

**データの永続化と管理.** 溶媒データベース、部品データ、ナノ粒子、医薬品、分散剤など複数の材料カテゴリのデータを安全に管理し、CRUD操作を提供する必要がある。

**クロスプラットフォーム.** 研究室のWindows PC、産業現場のLinuxワークステーション、macOS環境のいずれでも同一の動作を保証する必要がある。

## 19.2 三層アーキテクチャ

### 19.2.1 設計思想

本システムのアーキテクチャは、関心の分離（Separation of Concerns）を最上位の設計原則として、以下の三層に明確に分割される。

1. **メインプロセス層**（Electron Main Process）: Node.js環境で動作し、ファイルシステムアクセス・SQLiteデータベース操作・IPC通信のハブとして機能する
2. **レンダラープロセス層**（Electron Renderer Process）: Chromium環境で動作し、ReactコンポーネントとカスタムフックによるUI表示・ユーザーインタラクションを担当する
3. **コアドメインロジック層**（Core Domain Logic）: 純粋なTypeScript関数群からなり、HSP計算・バリデーション・型定義など、副作用のないドメインロジックをカプセル化する

[図19.1: 三層アーキテクチャ図 — Electronメインプロセス（Node.js、SQLite、IPCハンドラ）、レンダラープロセス（React、カスタムフック、Viewコンポーネント）、コアドメインロジック層（117の純粋関数モジュール）をブロック図で表現。contextBridgeを介したIPC通信とcoreからの直接インポートを矢印で示す]

この三層構造の最大の利点は、コアドメインロジック層がElectronにもReactにも依存しない点にある。`src/core/` 以下の117モジュールは純粋なTypeScript関数であり、Node.jsのfs・pathモジュールやReactのuseState・useEffectフックへの依存を持たない。これにより、メインプロセスからもレンダラープロセスからも安全にインポートでき、単体テストにおいてモック化が不要となる。

## 19.3 純粋関数によるドメインロジック設計

### 19.3.1 純粋関数の原則と利点

科学計算において「純粋関数」（pure function）を設計の核とすることは、極めて強力な品質保証戦略である。純粋関数とは、以下の二条件を同時に満たす関数を指す。

1. **参照透過性**: 同一の引数に対して常に同一の結果を返す
2. **副作用の不在**: 関数外部の状態を変更しない（グローバル変数の書き換え、ファイルI/O、データベースアクセス等を行わない）

HSP計算のような数値計算関数がこの原則に従うことで、以下の恩恵が得られる。

- **決定論的テスト**: 入力と期待出力の対を用意するだけで、完全なテストが可能
- **並列安全性**: 状態共有がないため、スレッドセーフティの問題が発生しない
- **コンポーザビリティ**: 小さな純粋関数を組み合わせて複雑な計算パイプラインを構築できる
- **文献検証**: 計算結果を文献値と直接照合でき、外部状態に起因する不一致が排除される

### 19.3.2 Ra・RED計算の設計例

本システムの`hsp.ts`モジュールは、純粋関数設計の典型例である。第1章で導入したHansen距離 $R_a$ の計算関数を以下に示す。

$$
R_a = \sqrt{4(\delta_{D1} - \delta_{D2})^2 + (\delta_{P1} - \delta_{P2})^2 + (\delta_{H1} - \delta_{H2})^2}
$$

> Unicode: Ra = sqrt[4(delta_D1 - delta_D2)^2 + (delta_P1 - delta_P2)^2 + (delta_H1 - delta_H2)^2]

```typescript
// src/core/hsp.ts — 純粋関数による Ra 計算
interface HSPValues {
  dD: number;  // 分散力成分 [MPa^0.5]
  dP: number;  // 極性成分 [MPa^0.5]
  dH: number;  // 水素結合成分 [MPa^0.5]
}

function calculateRa(a: HSPValues, b: HSPValues): number {
  return Math.sqrt(
    4 * (a.dD - b.dD) ** 2 +
    (a.dP - b.dP) ** 2 +
    (a.dH - b.dH) ** 2
  );
}

function calculateRed(ra: number, r0: number): number {
  if (r0 <= 0) throw new Error("R0 must be positive");
  return ra / r0;
}
```

この設計の要点は以下の通りである。

- **型安全性**: `HSPValues`インターフェースにより、三成分（dD, dP, dH）の受け渡しをコンパイル時に保証する
- **不変入力**: 引数は変更されず、新しい値が返される
- **明確なエラーハンドリング**: 無効な入力（$R_0 \leq 0$）に対して例外を投げる
- **テスト容易性**: 外部依存がないため、文献値との照合テストを即座に記述可能

### 19.3.3 バリデーションの純粋関数化

入力検証もまた純粋関数として設計される。バリデーション関数はエラー文字列またはnullを返す合成可能な関数として実装され、副作用を伴わない。

```typescript
// src/core/validation.ts — 合成可能なバリデータ
type Validator<T> = (value: T) => string | null;

function validatePositive(value: number, name: string): string | null {
  return value > 0 ? null : `${name}は正の値でなければなりません`;
}

function validateHSPValues(hsp: HSPValues): string | null {
  return (
    validatePositive(hsp.dD, "δD") ??
    validatePositive(hsp.dP, "δP") ??
    validatePositive(hsp.dH, "δH")
  );
}
```

モジュール間の結合度は、結合度係数（Coupling Factor）により定量化できる。

$$
CF = \frac{\sum_{i} d_{out}(i)}{N(N-1)}
$$

> Unicode: CF = Sigma d_out(i) / [N(N-1)]

ここで $d_{out}(i)$ はモジュール $i$ の出力依存数（他モジュールへの依存数）、$N$ はモジュール総数である。理想的なアーキテクチャではCFが低く保たれ、各モジュールが少数の明確な依存のみを持つ。本システムのコア層は、基盤モジュール（types.ts, hsp.ts）への集中的依存と、個別モジュール間の依存の最小化により、低い結合度を実現している。

## 19.4 Repositoryパターンによるデータアクセス

### 19.4.1 パターンの意義

Repositoryパターン [Fowler, 2002; Evans, 2003] は、ドメインロジックとデータアクセスの実装詳細を分離するための設計パターンである。科学計算ツールにおいてこのパターンを採用する利点は以下の通りである。

- **データベース技術の交換可能性**: SQLiteからPostgreSQLやファイルベースのJSONに変更しても、ドメインロジックは影響を受けない
- **テスト時のモック化**: インターフェースに基づくモックリポジトリを注入することで、データベースなしの高速テストが可能
- **クエリロジックの集約**: 特殊なクエリ（溶媒検索、カテゴリ別フィルタリング等）がリポジトリ実装に集約され、ドメインロジックの純粋性が維持される

### 19.4.2 リポジトリインターフェースの設計

本システムでは6つのリポジトリインターフェースが定義されており、それぞれが異なるデータエンティティのCRUD操作と特殊クエリを提供する。

[表19.4: リポジトリインターフェース一覧 — エンティティ型、CRUDメソッド、特殊クエリメソッド]

| リポジトリ | エンティティ型 | CRUDメソッド | 特殊クエリ |
|-----------|-------------|------------|-----------|
| `PartsRepository` | `Part` | create, read, update, delete | カテゴリ別一覧 |
| `SolventRepository` | `Solvent` | create, read, update, delete | `searchSolvents`（名称・CAS検索） |
| `SettingsRepository` | `Settings` | read, update | テーマ・閾値取得 |
| `NanoParticleRepository` | `NanoParticle` | create, read, update, delete | `getNanoParticlesByCategory` |
| `DrugRepository` | `Drug` | create, read, update, delete | 分類別フィルタ |
| `DispersantRepository` | `Dispersant` | create, read, update, delete | 用途別フィルタ |

### 19.4.3 SQLite実装とスキーママイグレーション

リポジトリの具象実装はbetter-sqlite3を用いたSQLiteアクセスである。スキーマ定義はSQLマイグレーションファイルとして管理され、アプリケーション起動時に自動適用される。

DTOから​​ドメインエンティティへのマッピング関数（row-to-domain mapping）を各リポジトリが保持することで、データベース行とドメインオブジェクトの変換責務がリポジトリ層に閉じ込められる。

### 19.4.4 シードデータアーキテクチャ

初期データの投入はドメイン固有のシードモジュール群により管理される。溶媒、ナノ粒子、医薬品、コーティング材、可塑剤、キャリア、分散剤の各カテゴリに専用シードモジュールが存在し、文献値に基づくHSPデータが体系的に登録される。このシードデータ設計により、新規カテゴリの追加が既存データに影響を与えることなく行える。

## 19.5 Electron IPC通信設計

### 19.5.1 セキュリティモデル

ElectronアプリケーションにおけるIPC（Inter-Process Communication）通信は、セキュリティ上の最重要課題である。レンダラープロセスがNode.js APIに直接アクセスできる場合、悪意あるスクリプトがファイルシステムやOSレベルの操作を実行し得る。このリスクを排除するため、本システムはElectronのcontextIsolation（コンテキスト分離）とcontextBridgeを厳格に適用する [Electron Contributors, 2013]。

contextBridgeはpreloadスクリプト内でのみ使用可能であり、レンダラープロセスに公開するAPIを明示的に定義する。レンダラーはwindow.apiオブジェクトを通じてのみメインプロセスと通信でき、Node.jsのfs、path、child_processなどのモジュールには一切アクセスできない。

### 19.5.2 通信フロー

IPC通信はipcMain.handle / ipcRenderer.invokeパターンを採用する。このリクエスト-レスポンスモデルにより、非同期通信がPromiseベースで型安全に行われる。

[図19.2: IPC通信フロー図 — ユーザーアクション → React Viewコンポーネント → カスタムフック → window.apiメソッド呼び出し → preload contextBridge → ipcRenderer.invoke → ipcMain.handle → コア純粋関数 → リポジトリデータアクセス → レスポンス逆順伝搬のシーケンス図]

通信フローの各段階を整理すると以下のようになる。

1. ユーザーがReact Viewコンポーネント上でアクションを実行する（例: 溶媒スクリーニング実行ボタンをクリック）
2. Viewコンポーネントに対応するカスタムフックが`window.api`のメソッドを呼び出す
3. preloadスクリプトのcontextBridgeが`ipcRenderer.invoke`を実行する
4. メインプロセスの`ipcMain.handle`がリクエストを受信する
5. IPCハンドラが適切なコア純粋関数を呼び出し、必要に応じてリポジトリからデータを取得する
6. 計算結果がレスポンスとして逆順に伝搬され、Viewコンポーネントの状態が更新される

### 19.5.3 型安全なIPC設計

TypeScript strict modeの下で、IPCチャネル名と引数型を型定義として管理することにより、メインプロセスとレンダラープロセスの間のインターフェース不整合をコンパイル時に検出する。チャネル名の文字列リテラル型やジェネリクスを活用し、invoke呼び出しとhandleハンドラの型が一致しない場合にはビルドエラーとなる設計が採用されている。

## 19.6 Reactフック・コンポーネントアーキテクチャ

### 19.6.1 カスタムフック設計

本システムのレンダラー層は80以上のカスタムフックを擁する。各フックは特定の評価パイプラインに対応し、以下の責務を一手に引き受ける。

- IPC呼び出しのカプセル化
- ローカル状態（入力値、計算結果、ローディング状態、エラー状態）の管理
- バリデーション結果の反映
- 副作用（データフェッチ、履歴保存等）の管理

このパターンにより、Viewコンポーネントは表示ロジックに専念でき、ビジネスロジックとの結合が最小化される。

### 19.6.2 Component-per-Viewパターン

93以上の評価タブの各々に専用のViewコンポーネントが対応する。各Viewは対応するカスタムフックから状態と操作関数を受け取り、入力フォーム・計算実行ボタン・結果テーブル・エラー表示などのUIを構成する。この1対1対応により、新しい評価パイプラインの追加は「コアモジュール → IPCハンドラ → preload公開 → カスタムフック → Viewコンポーネント → ナビゲーション登録」という一連の手順に還元され、既存コードへの影響が最小化される。

### 19.6.3 ナビゲーションアーキテクチャ

[図19.6: ナビゲーションアーキテクチャ — カテゴリベースのタブナビゲーションシステム。93タブがEvaluation、Screening、Visualization、Data Managementなどのカテゴリに分類され、各タブがViewコンポーネント+カスタムフックにマッピングされるツリー図]

93のタブはカテゴリベースのルーティングシステムにより組織化される。各カテゴリ（評価、スクリーニング、可視化、データ管理等）の下にタブが配置され、ユーザーは階層的なナビゲーションにより目的の評価パイプラインに到達する。新しいタブの追加はナビゲーション設定ファイルへのエントリ追加で完了し、ルーティングロジック自体の変更は不要である。

### 19.6.4 エラーバウンダリ

Reactの`ErrorBoundary`コンポーネントにより、個別タブでのランタイムエラーがアプリケーション全体のクラッシュに波及することを防止する。エラー発生時にはフォールバックUIが表示され、ユーザーは他のタブでの作業を継続できる。科学計算ツールにおいて、特定の計算で例外が発生してもツール全体が使用不能にならないことは、実用上極めて重要である。

## 19.7 テスト駆動開発とテストピラミッド

### 19.7.1 テスト駆動開発の方法論

テスト駆動開発（TDD）[Beck, 2002] は「RED → GREEN → REFACTOR」のサイクルを反復する開発手法である。科学計算ソフトウェアにおいてTDDが特に有効である理由は以下の通りである。

- **文献値との照合が先行**: 文献に記載されたHSP値・Ra値・RED値を期待値としてテストを記述し、その後に実装を行うことで、計算の正確性が設計段階で保証される
- **回帰テストの自動化**: 最適化やリファクタリング後に全テストを再実行することで、既存計算の正確性が維持されていることを自動確認できる
- **境界条件の網羅**: TDDでは正常系だけでなく、ゼロ除算、負の入力、NaN値などの境界条件を最初から考慮することが強制される

### 19.7.2 テストピラミッド構造

[図19.5: 科学技術ソフトウェアのテストピラミッド — 底辺に2604件のユニットテスト（Vitest）でコア計算とバリデーションをカバー、中間層に統合テスト（リポジトリ+シードデータ）、頂点に295件のE2Eテスト（Playwright）で93タブにわたるユーザーワークフローをカバーするピラミッド図]

本システムのテスト戦略は古典的なテストピラミッドに従う。

**底辺: ユニットテスト（2604件、Vitest）.** コアドメインロジックの純粋関数を網羅的にテストする。`calculateRa`に対する文献値照合、バリデーション関数のエラーケース、混合則の数値精度などが含まれる。純粋関数であるため、モック化は不要であり、テストの実行速度は極めて高速である。

**中間層: 統合テスト.** リポジトリ実装とシードデータの整合性をテストする。SQLiteデータベースの生成・マイグレーション・シードデータ投入・CRUD操作の一連のフローを検証する。

**頂点: E2Eテスト（295件、Playwright）.** ユーザーの操作シナリオに沿って、タブ切り替え・入力・計算実行・結果確認のフローを93タブにわたって検証する。Electronアプリケーションの実際のUIを操作するため、IPC通信・データベースアクセス・React描画を含む全スタックが検証対象となる。

### 19.7.3 カバレッジメトリクス

コードカバレッジは以下の式で定義される。

$$
C = \frac{L_{executed}}{L_{total}} \times 100\%
$$

> Unicode: C = L_executed / L_total x 100%

ここで $L_{executed}$ はテストスイートにより実行されたソースコード行数、$L_{total}$ は全ソースコード行数である。本システムは $C = 98.88\%$（2604件のテスト）を達成している。

[表19.5: テストカバレッジサマリー — アーキテクチャ層別のテスト件数とカバレッジ]

| テスト層 | テスト件数 | テストフレームワーク | カバレッジ/備考 |
|---------|----------|-------------------|---------------|
| コアユニットテスト | ~2000件 | Vitest | 98.88%（行カバレッジ） |
| バリデーションテスト | ~300件 | Vitest | 全バリデータの正常・異常系 |
| 統合テスト | ~300件 | Vitest | リポジトリ+シードデータ整合性 |
| E2Eテスト | 295件 | Playwright | 93タブの主要ユーザーフロー |

### 19.7.4 高カバレッジの功罪

98.88%というカバレッジ目標は、科学計算の正確性要求に応えるものであるが、盲目的な100%追求には注意が必要である [McConnell, 2004]。UIのスタイリングコードや定数定義ファイルなど、テストの費用対効果が低い箇所が存在する。また、カバレッジはコードの「実行」を保証するが「正確性」を保証するものではない。文献値との照合、境界値分析、プロパティベーステストなど、カバレッジ数値に表れない品質保証手法との併用が不可欠である。

## 19.8 複雑度管理

ソフトウェアの保守性を定量的に評価するうえで、McCabeのサイクロマティック複雑度 [McCabe, 1976] は広く用いられる指標である。

$$
M = E - N + 2P
$$

> Unicode: M = E - N + 2P

ここで $E$ はプログラムの制御フローグラフにおけるエッジ数、$N$ はノード数、$P$ は連結成分数である。$M$ が高い関数は分岐が多く、テストケース数が増大し、バグの温床となりやすい。

純粋関数設計は複雑度の低減に直接寄与する。外部状態への依存がないため、条件分岐が入力パラメータのバリデーションに限定される傾向がある。例えば`calculateRa`の $M$ 値は1（分岐なし）であり、`calculateRed`は2（$R_0 \leq 0$ のガード節による分岐1つ）である。各純粋関数の $M$ 値を10以下に保つことが、本プロジェクトの設計基準として運用されている。

## 19.9 拡張性の設計 — 新しい評価パイプラインの追加

本アーキテクチャの真価は、新しい評価パイプラインを追加する際の手順の定型性と既存コードへの影響の最小性に表れる。以下に、新しいHSP評価機能（例: ポリマー適合性評価）を追加する際のステップを示す。

**ステップ1: コア純粋関数モジュールの作成.** `src/core/polymer-compatibility.ts` に型付きの入出力を持つ純粋関数を実装する。既存の`hsp.ts`（Ra/RED計算）や`types.ts`（HSPValues型）を再利用する。

**ステップ2: バリデーション関数の追加.** `validation.ts` に入力バリデータを追加するか、専用バリデーションモジュールを作成する。

**ステップ3: IPCハンドラの登録.** `ipc-handlers.ts` に新しいハンドラを登録し、コア関数を呼び出すルーティングを追加する。

**ステップ4: preload contextBridgeでのAPI公開.** `preload.ts` に新しいAPIメソッドを追加し、レンダラーに公開する。

**ステップ5: カスタムReactフックの作成.** `usePolymerCompatibility` のようなカスタムフックを作成し、IPC呼び出しと状態管理をカプセル化する。

**ステップ6: Viewコンポーネントの構築.** `PolymerCompatibilityView.tsx` を作成し、入力フォーム・実行ボタン・結果テーブルのUIを構成する。

**ステップ7: ナビゲーション登録.** ナビゲーション設定に新しいタブエントリを追加する。

**ステップ8: テストの記述.** ユニットテスト（コア関数の文献値照合）とE2Eテスト（タブ操作→計算実行→結果確認）を記述する。

この8ステップにおいて、既存のコアモジュール・リポジトリ・他のViewコンポーネントへの変更は不要である。`ipc-handlers.ts`、`preload.ts`、ナビゲーション設定への追記のみが既存ファイルへの変更となり、変更の影響範囲が明確に制限される。

## 章末問題

**問題19.1.** 「contextBridgeを用いないElectronアプリケーション」と「contextBridgeを適用したアプリケーション」のセキュリティリスクの差異を論述せよ。レンダラープロセスがNode.js APIに直接アクセスできる場合に生じ得る具体的な攻撃シナリオ（例: リモートコード実行）を挙げて説明すること。

**問題19.2.** 本プロジェクトのRepositoryパターン（インターフェース + SQLite実装）と、他のデータアクセスパターン（Active Record、Data Mapper）を比較せよ。基盤データベース技術が変更される可能性がある場合の利点を、レガシーコードの保守 [Feathers, 2004] の観点から議論すること。

**問題19.3.** 本プロジェクトは2604件のユニットテストで98.88%のコードカバレッジを達成している。科学計算ソフトウェアにおける高カバレッジ目標と開発速度のトレードオフを議論せよ。100%カバレッジが逆効果となるケースを具体例とともに示し、カバレッジ以外の品質指標（ミューテーションテスト、プロパティベーステスト等）の役割を論じること。

**問題19.4.** 新しい評価パイプライン（例: 新規ポリマー適合性評価）をこのアーキテクチャに追加する手順を詳述せよ。どのファイルを新規作成し、どのファイルを修正する必要があるか。モジュラー設計が既存コードへの影響をいかに最小化するかを、結合度係数 $CF$ の観点から説明すること。

**問題19.5.** HSP計算関数（$R_a$, RED）を純粋関数として実装することの重要性を、テスト容易性と信頼性の観点から論じよ。副作用を持つ実装（グローバル変数への依存、データベースアクセスの混入等）がもたらす具体的な問題を3つ以上挙げ、純粋関数設計がそれぞれをどのように解決するかを示すこと。
