# 第18章 データ管理と品質保証 — CSVインポート、データベース設計、レポート生成

**Chapter 18: Data Management and Quality Assurance — CSV Import, Database Design, and Report Generation**

> **概要:** 本章では、HSP評価の信頼性を支えるデータインフラストラクチャを体系的に解説する。第一に、CSVファイルによる溶媒・材料データのインポート/エクスポートと行レベルバリデーションの設計を導入する。第二に、SQLiteデータベースによる永続的データ管理——溶媒、ポリマー、ナノ粒子、薬物、分散剤の5エンティティに対するスキーマ設計とリポジトリパターンの適用——を詳述する。第三に、200件を超えるシードデータの構築と管理手法、ブックマーク・評価履歴システムによる再現性の確保を論じる。最後に、既知の推定バイアスに基づく精度警告メカニズム、およびPDF・CSVレポート生成による規制対応文書化を解説する。

## 18.1 HSPデータ管理の歴史的展開

### 18.1.1 手作業からデジタルデータベースへ

HSPデータ管理の歴史は、溶解度パラメータ理論そのものの発展と軌を一にしている。Charles Hansenが1967年に三成分HSPを提唱した際、溶媒パラメータは手書きの表で管理されていた。初期のデータセットは限られた溶媒群に対する実験的溶解試験の結果を集約したものであり、数十〜百件程度の規模であった。

最初の包括的デジタルHSPデータベースは、Hansen and Beerbower（1971）による Kirk-Othmer Encyclopedia への寄稿に含まれ、約200種の溶媒を収録していた [Hansen & Beerbower, 1971]。このデータベースは、δ_D、δ_P、δ_H の三成分値に加え、モル体積や沸点などの基本物性を併記するフォーマットを確立した。

Barton（1991）は CRC Handbook of Solubility Parameters として、溶解度パラメータに関する最も包括的な文献データ集を出版した [Barton, 1991]。この著作は実験値と推算値を体系的に整理し、データの出典と推定精度に関する情報を付記するという品質管理の先例を示した。

### 18.1.2 HSPiPとデータ管理の標準化

Abbott and Hansen（2008）が開発したHSPiP（Hansen Solubility Parameters in Practice）は、1200種を超える溶媒データベースを搭載し、CSVインポート/エクスポートワークフローを初めて標準化した [Abbott & Hansen, 2008]。HSPiPの登場により、企業や研究機関が自社測定データをCSV形式でインポートし、既存データベースと統合するという実務フローが確立された。

### 18.1.3 現代のデータ管理要件

現代のHSP評価ツールに求められるデータ管理機能は、以下の要件を含む。

- **多様なエンティティ管理**: 溶媒のみならず、ポリマー、ナノ粒子、薬物、分散剤を統一的に管理する必要がある
- **データ品質保証**: インポート時の自動バリデーション、既知バイアスへの警告生成
- **再現性**: 評価パラメータのブックマーク保存と正確な復元
- **規制対応**: GLP（Good Laboratory Practice）[OECD, 1998] に準拠した文書化・監査証跡
- **相互運用性**: CSVによる外部システムとのデータ交換、PDF報告書の生成

これらの要件を満たすデータ管理アーキテクチャの設計が、本章の主題である。

## 18.2 CSVインポートとバリデーション

### 18.2.1 CSVフォーマット設計

HSPデータのCSVインポートにおいて、カラムマッピングの設計は相互運用性の鍵となる。溶媒インポートと材料（パーツ）インポートでは、必須・任意カラムの構成が異なる。

[表18.1: CSVカラムマッピング — 溶媒インポートでは name, deltaD, deltaP, deltaH が必須、nameEn, casNumber, boilingPoint, viscosity, specificGravity, surfaceTension, molarVolume, molWeight が任意。材料インポートでは name, deltaD, deltaP, deltaH, r0 が必須、materialType が任意]

溶媒インポートのCSVフォーマット例を以下に示す。

```csv
name,deltaD,deltaP,deltaH,boilingPoint,viscosity,casNumber
トルエン,18.0,1.4,2.0,110.6,0.590,108-88-3
アセトン,15.5,10.4,7.0,56.1,0.320,67-64-1
エタノール,15.8,8.8,19.4,78.3,1.200,64-17-5
```

材料インポートでは相互作用半径 $R_0$ が必須カラムに加わる点が重要である。$R_0$ はHSP球の半径を定義し、溶解性判定の閾値となるため、欠損は許容されない。

### 18.2.2 行レベルバリデーション戦略

CSVインポートのバリデーション設計において、最も重要な設計判断は「ファイル全体の拒否」と「行レベルの検証」のいずれを採用するかである。本システムでは行レベルバリデーションを採用する。この設計判断の根拠は以下の通りである。

1. **部分インポートの実用性**: 数百行のCSVファイルにおいて、1行のエラーで全体を棄却すると、ユーザーの修正コストが不必要に増大する
2. **エラー特定の容易性**: 行番号とフィールド名を特定したエラーメッセージにより、修正箇所が明確になる
3. **段階的データ移行**: 既存データベースからの移行時に、フォーマット不整合のある行を特定しつつ有効データを取り込める

バリデーションの各項目を以下に整理する。

| 検証項目 | 条件 | エラーメッセージ例 |
|---------|------|-----------------|
| name（名称） | 空文字列でないこと | "行3: 名称が空です" |
| deltaD | 数値であること | "行5: δ_D が数値ではありません" |
| deltaP | 数値であること | "行5: δ_P が数値ではありません" |
| deltaH | 数値であること | "行7: δ_H が数値ではありません" |
| r0（材料のみ） | 正の数値であること | "行4: R₀ は正の値が必要です" |

[図18.1: CSVインポートバリデーションパイプラインフローチャート — ファイル読み込み → ヘッダー抽出 → 行反復 → フィールドレベルバリデーション（名称非空、δ_D/δ_P/δ_H 数値、R₀正値）→ エラー収集 → ImportResult（有効行リスト + エラーメッセージリスト）出力の流れを示す]

### 18.2.3 例題：バリデーションエラー処理を伴うCSVインポート

以下のCSVファイルを考える。20行の溶媒データのうち、意図的に不正な行を含むものである。

```csv
name,deltaD,deltaP,deltaH,boilingPoint
トルエン,18.0,1.4,2.0,110.6
,15.5,10.4,7.0,56.1
エタノール,abc,8.8,19.4,78.3
MEK,16.0,9.0,5.1,79.6
```

行レベルバリデーションを適用すると、以下の結果が得られる。

- **行1（トルエン）**: 全フィールド有効 → 有効行として取り込み
- **行2**: name が空 → エラー "行2: 名称が空です"
- **行3（エタノール）**: deltaD が "abc"（非数値）→ エラー "行3: δ_D が数値ではありません"
- **行4（MEK）**: 全フィールド有効 → 有効行として取り込み

最終的な `ImportResult` は、有効行 2件（トルエン、MEK）とエラーメッセージ 2件を含む。ファイル全体が棄却されることなく、有効データが利用可能となる。

### 18.2.4 BOM付きUTF-8 CSVエクスポート

日本語環境でのExcel互換性を確保するため、CSVエクスポートではBOM（Byte Order Mark）付きUTF-8エンコーディングを使用する。BOMは3バイトのプレフィックス（`0xEF, 0xBB, 0xBF`）であり、ExcelがファイルをUTF-8として正しく認識するために必要である。

BOMなしのUTF-8 CSVをExcelで開くと、日本語文字が文字化けする問題はHSPデータ管理において頻繁に遭遇する実務的課題であり、BOMプレフィックスの付加はこの問題に対する標準的な解決策である。

## 18.3 データベース設計

### 18.3.1 SQLiteとWALジャーナリングモード

本システムのデータベースエンジンとしてSQLiteを採用する。SQLiteは組み込み型RDBMSであり、サーバープロセスを必要とせず、単一ファイルでデータベースを管理できる。HSP評価ツールのようなデスクトップアプリケーションにおいて、この特性は運用コストの観点から大きな利点となる。

WAL（Write-Ahead Logging）ジャーナリングモードを有効化することで、以下の並行性が実現される。

- **複数の同時読み取り**: WALモードでは、読み取りトランザクションが書き込みトランザクションをブロックしない
- **書き込み中の読み取り**: 書き込みが進行中であっても、読み取りは直前のコミット済みデータを参照できる
- **マルチサイト展開**: 共有データベースファイルへの同時読み取りアクセスが可能となる

外部キー制約を明示的に有効化（`PRAGMA foreign_keys = ON`）することで、エンティティ間の参照整合性を保証する。

### 18.3.2 エンティティ・リレーションシップ設計

HSP評価システムのデータモデルは、以下のエンティティで構成される。

[図18.2: データベースER図 — solvents, parts/parts_groups, nano_particles, drugs, dispersants, bookmarks, evaluation_history, settings テーブルを示すER図。外部キー関係とインデックス注記を含む]

各テーブルの主要仕様を以下に整理する。

[表18.2: データベーステーブル仕様 — 全SQLiteテーブルのカラム名、型、制約、デフォルト値。solvents（15カラム）、parts_groups（4カラム）、parts（10カラム）、nano_particles（12カラム）、drugs（12カラム）、dispersants（19カラム）、bookmarks（5カラム）、evaluation_history（7カラム、インデックス2個）、settings（2カラム）]

**solvents テーブル**: 溶媒の基本HSP値（$\delta_D$, $\delta_P$, $\delta_H$）に加え、沸点、粘度、比重、表面張力、モル体積、分子量、CAS番号を格納する。15カラムの構成は、HSPiPのデータフォーマットとの互換性を考慮した設計である。

**parts / parts_groups テーブル**: ポリマー（材料）は、parts_groups テーブルによる階層的グループ化を持つ。汎用プラスチック、エンジニアリングプラスチック、スーパーエンジニアリングプラスチック、熱硬化性樹脂、ゴム・エラストマー、フッ素樹脂、接着剤の7グループが定義される。各 part は $\delta_D$, $\delta_P$, $\delta_H$ に加え、相互作用半径 $R_0$ を必須フィールドとして持つ。

**nano_particles テーブル**: ナノ粒子は表面修飾の有無、粒子径など12カラムの情報を持ち、第11章（ナノ材料分散）と連携する。

**drugs テーブル**: 薬物は12カラムの情報を持ち、第9章（医薬品ドラッグデリバリー）の評価パイプラインで使用される。

**dispersants テーブル**: 分散剤は19カラムと最多のフィールドを持ち、HLB値や分散対象などの分散剤固有の情報を含む。

### 18.3.3 リポジトリパターン

データアクセス層の設計にはリポジトリパターン [Fowler, 2002] を採用する。このパターンは、データアクセスのインターフェースとSQLite実装を分離する。

[図18.3: リポジトリパターンアーキテクチャ図 — PartsRepository, SolventRepository, NanoParticleRepository, DrugRepository, DispersantRepository のインターフェースとそのSQLite実装をUML形式で示す。DTO型（create/update操作用）を含む]

リポジトリパターンの主要な利点は以下の通りである。

1. **テスタビリティ**: インターフェースに対するモック実装を用いて、データベースなしでビジネスロジックの単体テストが可能となる
2. **実装交換性**: SQLiteからPostgreSQLや他のRDBMSへの将来的な移行時に、インターフェースを満たす新たな実装を追加するだけでよい
3. **関心の分離**: ビジネスロジック層はSQL文の詳細を知る必要がなく、リポジトリインターフェースを通じてデータにアクセスする

各リポジトリは、CRUD操作（Create, Read, Update, Delete）に対応するメソッドを定義し、DTO（Data Transfer Object）型を用いてcreate/update操作のパラメータを型安全に受け渡す。

## 18.5 精度警告システム

### 18.5.1 設計思想

HSP推定値には既知の系統的バイアスが存在し、これをユーザーに適切に伝達することは品質保証の重要な要素である。精度警告システムは、147件の文献バリデーション試験から同定された系統的偏差パターンに基づき、評価結果に対して自動的に警告を生成する。

警告は「誤差の存在を通知する」ことが目的であり、評価結果を棄却するものではない。ユーザーが結果の解釈に際して、既知のバイアスの方向と大きさを認識した上で判断できるよう支援する。

### 18.5.2 警告カテゴリと閾値

[表18.4: 精度警告の閾値と偏差量 — アルコール表面張力過大推定（最大 +13 mN/m）、ポリオール過小推定（最大 -10 mN/m）、親水性ポリマー水接触角過大推定（最大 +17度）、RED境界曖昧域（0.8-1.2）、ナノ粒子 R₀ 文献散布（±1.0）、量子サイズ効果閾値（<5 nm）]

各警告カテゴリの詳細を以下に述べる。

**アルコールプロファイル検出**（$\delta_H > 14$ かつ $\delta_P > 5$）: 中村-山本（Nakamoto-Yamamoto）式による表面張力推定は、アルコール類に対して系統的に過大推定する傾向がある。これは水素結合成分が高い溶媒において、分子表面での水素結合ネットワーク形成が推定モデルの仮定と乖離するためである。最大偏差は +13 mN/m に達する。

**ポリオールプロファイル検出**（$\delta_H > 20$ かつ $\delta_P > 10$）: ポリオール（多価アルコール）に対しては逆に過小推定が生じ、最大 -10 mN/m の偏差が報告されている。

**親水性ポリマーの水接触角**: 親水性表面と水の組み合わせにおいて、接触角推定は最大 +17度の過大推定を示す。水の異常に高い水素結合パラメータ（$\delta_H > 35$ MPa$^{0.5}$）が、推定モデルの線形近似の範囲を超えることが主因である。

**RED境界曖昧域**（0.8 < RED < 1.2）: RED = 1.0 は溶解性の理論的境界であるが、実際にはこの近傍の結果は確率的解釈が必要である。0.8〜1.2の範囲にある結果に対しては、「境界付近であり実験的確認を推奨」との警告を生成する。

**ナノ粒子関連警告**: ナノ粒子のHSPおよび $R_0$ は文献値の散布が大きく（$\pm 1.0$ MPa$^{0.5}$）、表面修飾の影響も顕著である。粒径 5 nm 未満の場合は量子サイズ効果により、バルク物性からの推定が不適切となる可能性がある。

[図18.4: 精度警告判定ツリー — アルコールプロファイル検出（δ_H>14, δ_P>5）、ポリオールプロファイル（δ_H>20, δ_P>10）、水検出（δ_H>35）、親水性ポリマー判定、RED境界域（0.8-1.2）、ナノ粒子表面修飾/量子サイズ効果の条件分岐を示す決定木図]

### 18.5.3 例題：接触角評価における精度警告生成

エタノール（アルコールプロファイル: $\delta_H = 19.4$, $\delta_P = 8.8$）を用いて、親水性ポリマーと水系での接触角評価を実施する場合を考える。

**判定プロセス:**

1. **アルコール検出**: $\delta_H = 19.4 > 14$ かつ $\delta_P = 8.8 > 5$ → アルコールプロファイル検出
   - 警告: "アルコール系溶媒のため、表面張力推定値は最大 +13 mN/m 過大となる可能性があります"
2. **ポリオール検出**: $\delta_H = 19.4 < 20$ → ポリオールには該当しない
3. **親水性ポリマー + 水の組み合わせ**: 対象が親水性ポリマーで水との接触角を評価する場合
   - 警告: "親水性ポリマーの水接触角推定は最大 +17度の過大推定を含む可能性があります"

合計2件の警告が生成され、ユーザーは結果の解釈に際してこれらのバイアスを考慮できる。

## 18.6 MD シミュレーションCEDインポート

### 18.6.1 CED-HSP変換

分子動力学（MD）シミュレーションから得られる凝集エネルギー密度（CED: Cohesive Energy Density）は、HSP成分に変換できる。CED成分から各HSP成分への変換式は以下の通りである [Belmares et al., 2004]。

$$
\delta_i = \sqrt{\text{CED}_i}
$$

> Unicode: δ_i = sqrt(CED_i)  [i = D, P, H]

ここで CED_i の単位は J/cm³ = MPa であり、$\delta_i$ の単位は MPa$^{0.5}$ となる。

### 18.6.2 整合性チェック

MDシミュレーションでは、全CED（totalCED）と成分CED（dispersionCED, polarCED, hbondCED）が独立に計算される場合がある。これらの整合性を検証するため、以下の整合性指標を定義する。

$$
\text{Consistency} = \left(1 - \frac{|\delta_{\text{total}} - \sqrt{\delta_D^2 + \delta_P^2 + \delta_H^2}|}{\max(\delta_{\text{total}}, \sqrt{\delta_D^2 + \delta_P^2 + \delta_H^2})}\right) \times 100\%
$$

> Unicode: Consistency = (1 − |δ_total − sqrt(δ_D² + δ_P² + δ_H²)| / max(δ_total, sqrt(δ_D² + δ_P² + δ_H²))) × 100%

整合性が90%を下回る場合、成分分解の妥当性に疑義が生じる。原因として以下が考えられる。

- シミュレーション時間が不十分（平衡未到達）
- 力場パラメータの不整合
- 長距離相互作用のカットオフ設定
- クロスターム（分散-極性、極性-水素結合のカップリング）の寄与

### 18.6.3 例題：トルエンのMD CED インポート

トルエンのMDシミュレーションから以下のCED成分が得られたとする。

| CED成分 | 値 (MPa) |
|---------|---------|
| dispersionCED | 324 |
| polarCED | 1.96 |
| hbondCED | 4.0 |
| totalCED | 330 |

**HSP計算:**

$$
\delta_D = \sqrt{324} = 18.0 \text{ MPa}^{0.5}
$$

$$
\delta_P = \sqrt{1.96} = 1.4 \text{ MPa}^{0.5}
$$

$$
\delta_H = \sqrt{4.0} = 2.0 \text{ MPa}^{0.5}
$$

$$
\delta_{\text{total}} = \sqrt{330} = 18.17 \text{ MPa}^{0.5}
$$

**整合性チェック:**

成分ベクトル和:

$$
\sqrt{\delta_D^2 + \delta_P^2 + \delta_H^2} = \sqrt{18.0^2 + 1.4^2 + 2.0^2} = \sqrt{324 + 1.96 + 4.0} = \sqrt{329.96} = 18.165 \text{ MPa}^{0.5}
$$

$$
\text{Consistency} = \left(1 - \frac{|18.17 - 18.165|}{\max(18.17, 18.165)}\right) \times 100\% = \left(1 - \frac{0.005}{18.17}\right) \times 100\% \approx 99.97\%
$$

整合性 99.97% であり、成分分解は極めて良好である。得られたHSP値（$\delta_D = 18.0$, $\delta_P = 1.4$, $\delta_H = 2.0$）は文献値 [Hansen, 2007] とも良好に一致する。

**物理的妥当性検証**: $\delta_D$, $\delta_P$, $\delta_H$ がいずれも正値であり、トルエンの既知物性（非極性芳香族溶媒）と矛盾しないことを確認する。$\delta_D$ が支配的で $\delta_P$, $\delta_H$ が小さいことは、分散力主体の分子間相互作用を正しく反映している。

## 18.7 混合溶媒のHSP計算

CSVインポートされた溶媒を用いてブレンド計算を行う場合、混合溶媒のHSP値は体積分率加重の線形混合則により計算される。

$$
\delta_{\text{mix},i} = \sum_k \phi_k \, \delta_{k,i}
$$

> Unicode: δ_mix,i = Σ_k φ_k · δ_k,i

ここで $\phi_k$ は溶媒 $k$ の体積分率、$\delta_{k,i}$ は溶媒 $k$ のHSP成分 $i$（$i = D, P, H$）である。

この線形混合則は、溶媒間の相互作用が理想的である（混合のエンタルピー変化がゼロに近い）場合に良好な近似を与える。強い水素結合を形成する溶媒の混合（例：水-アルコール系）では、非理想性による偏差が生じることがあり、第16章で論じた高度なブレンド最適化手法を参照されたい。

## 18.8 ブックマークと評価履歴

### 18.8.1 ブックマークシステム

HSP評価の再現性を確保するため、評価パラメータをブックマークとして保存・復元するシステムを実装する。ブックマークは以下の情報を保持する。

- **パイプライン種別**: 9種の評価パイプライン（溶解リスク評価、接触角推定、膨潤予測、耐薬品性、ナノ分散、可塑剤選定、キャリア選定、ブレンド最適化、薬物溶解性）のいずれか
- **評価パラメータ**: 各パイプラインに固有のパラメータセット（対象材料HSP、候補溶媒リスト、評価条件など）をJSON形式で直列化
- **タイムスタンプ**: ブックマーク作成日時
- **名称**: ユーザー定義の識別名

### 18.8.2 評価履歴

評価履歴リポジトリは、実行された全評価を自動的に記録する。各エントリは以下の情報を含む。

- パイプライン種別
- 直列化されたパラメータ（JSON）
- 直列化された結果（JSON）
- 実行日時
- 実行所要時間

**自動プルーニング**: 評価履歴はFIFO（First-In, First-Out）方式で最大1000件に制限される。1000件を超える場合、最も古いエントリから自動的に削除される。

[図18.5: 評価履歴とブックマークのデータフロー — ユーザー評価 → JSON直列化（JSON.stringify）→ SQLite保存 → 取得 → 逆直列化 → 再実行（ブックマーク）の流れと、1000件リミットでの自動プルーニングメカニズム（評価履歴）を示すシーケンス図]

### 18.8.3 例題：再現可能な評価のブックマーク・履歴ワークフロー

ナノ粒子分散評価のブックマークワークフローを以下に示す。

**ステップ1: 評価実行とブックマーク保存**

ナノ粒子（TiO₂, $\delta_D = 19.3$, $\delta_P = 6.6$, $\delta_H = 11.3$, $R_0 = 4.5$）に対する分散溶媒スクリーニングを実行する。評価パラメータをJSON直列化してブックマークに保存する。

```json
{
  "pipeline": "nanoDispersion",
  "params": {
    "particle": { "deltaD": 19.3, "deltaP": 6.6, "deltaH": 11.3, "r0": 4.5 },
    "candidateSolvents": ["トルエン", "MEK", "エタノール", "酢酸エチル"]
  },
  "name": "TiO2分散評価_2026-03-30",
  "createdAt": "2026-03-30T10:00:00Z"
}
```

**ステップ2: ブックマークからの復元**

保存されたブックマークをSQLiteから取得し、JSONを逆直列化してパラメータを復元する。復元されたパラメータで評価を再実行すると、同一の結果が得られる。

**ステップ3: 評価履歴の確認**

評価履歴リストには、直近の評価結果がタイムスタンプとともに記録されている。1000件を超えた場合のプルーニング動作を確認するため、大量評価のテストでは最古のエントリが自動削除されることを検証する。

> **GLP/GMP対応に関する注意**: 規制環境では、評価履歴の自動プルーニングが監査証跡の完全性を損なう可能性がある。コンプライアンス要件が厳しい場合は、プルーニング上限の引き上げ、外部アーカイブシステムへの定期エクスポート、またはプルーニング無効化の選択肢を検討すべきである。
